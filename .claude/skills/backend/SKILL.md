# Back-end Engineer Skill

You write the server. Your work powers the interactive AI features of the dashboard — Generate Brief, Generate Reply, Regenerate, Suggest Tasks, Ask. The display data (calendar, inbox, tasks, notion) is baked into `dashboard.html` at generation time by the frontend-engineer using MCP data, so the server does NOT fetch Gmail or Calendar data. It only calls the Anthropic API.

## The architecture you build for

**Hybrid: static dashboard + tiny interactive server.**

- The dashboard HTML is a static file generated by the frontend-engineer with all the data (calendar events, emails, tasks) baked in.
- Your server runs locally on `http://localhost:3001` and exposes a small set of API endpoints that the dashboard's JavaScript calls when the user clicks an AI-powered button.
- Each endpoint takes the relevant data (e.g., a meeting object, an email body) in the request body, calls the Anthropic API, and returns the AI's response.
- No Gmail or Calendar API calls. No OAuth. No `googleapis`. No `mcp_servers` parameter. Just plain Anthropic API calls.

## Tech stack you use

- **Node.js** — runtime
- **Express** (latest 4.x) — server framework
- **@anthropic-ai/sdk** (latest) — for Anthropic API calls
- **dotenv** (latest) — for loading the API key from `.env`
- **cors** — to handle cross-origin requests from the dashboard if needed

Nothing else unless the task explicitly requires it.

### `server.js`

The main server. Structure:

```javascript
require('dotenv').config();
const express = require('express');
const cors = require('cors');
const Anthropic = require('@anthropic-ai/sdk');

const app = express();
const port = 3001;

app.use(cors());
app.use(express.json({ limit: '10mb' }));
app.use(express.static(__dirname));

// Verify API key on startup
if (!process.env.ANTHROPIC_API_KEY) {
  console.error('ERROR: ANTHROPIC_API_KEY not found in .env file.');
  console.error('Create a .env file in this folder with: ANTHROPIC_API_KEY=sk-ant-api03-...');
  process.exit(1);
}

const anthropic = new Anthropic({ apiKey: process.env.ANTHROPIC_API_KEY });

// Helper to call Claude with the user's CLAUDE.md as system context
async function callClaude(systemPrompt, userMessage, maxTokens = 1024) {
  try {
    const response = await anthropic.messages.create({
      model: 'claude-sonnet-4-6',
      max_tokens: maxTokens,
      system: systemPrompt,
      messages: [{ role: 'user', content: userMessage }],
    });
    return response.content[0].text;
  } catch (error) {
    console.error('Anthropic API error:', error.message);
    throw error;
  }
}

// Helper to load CLAUDE.md
function loadClaudeMd() {
  const fs = require('fs');
  const path = require('path');
  try {
    return fs.readFileSync(path.join(__dirname, 'CLAUDE.md'), 'utf-8');
  } catch (error) {
    return 'No CLAUDE.md profile found.';
  }
}

// Serve dashboard.html at the root
app.get('/', (req, res) => {
  res.sendFile(__dirname + '/dashboard.html');
});

// Health check
app.get('/api/health', (req, res) => {
  res.json({ status: 'ok' });
});

// [Feature endpoints get added here in later phases]

app.listen(port, () => {
  console.log(`✓ Server running on http://localhost:${port}`);
  console.log(`  Open this URL in your browser to see your dashboard.`);
});
```

## Feature phases — building API endpoints

For each feature phase, you add ONE endpoint to `server.js`. Each endpoint follows the same pattern:

```javascript
app.post('/api/[feature-name]', async (req, res) => {
  try {
    const { /* destructure inputs from req.body */ } = req.body;

    // Build the system prompt — always includes CLAUDE.md
    const claudeMd = loadClaudeMd();
    const systemPrompt = `${claudeMd}\n\n[Any task-specific instructions]`;

    // Build the user message with the relevant data
    const userMessage = `[Description of the task + the data]`;

    // Call Claude
    const answer = await callClaude(systemPrompt, userMessage);

    res.json({ /* structured response */ });
  } catch (error) {
    res.status(500).json({ error: 'AI feature temporarily unavailable. Try again in a moment.' });
  }
});
```

### Specific endpoints by feature

**Calendar / Generate Brief — `POST /api/meeting/brief`**
- Input: `{ meeting: { title, time, location, attendees: [{name, email}] } }`
- Output: `{ companyContext: string, suggestedQuestions: [string] }`
- Prompt: ask Claude to infer company context from attendee email domains and generate 3-4 questions grounded in the user's CLAUDE.md profile.

**Email / Generate Reply — `POST /api/email/reply`**
- Input: `{ email: { sender, subject, body } }`
- Output: `{ draftReply: string, sourceFiles: [string] }`
- Prompt: ask Claude to read any files in `documents/` that look relevant, then draft a reply that matches the user's CLAUDE.md tone. NEVER invent names, dates, numbers, or commitments. Flag gaps with `[need detail: X]`.

**Email / Regenerate Reply — same endpoint, called again**
- The frontend just calls `/api/email/reply` a second time. Add a slight `temperature: 0.8` if the SDK supports it to encourage variation.

**Tasks / Suggest — `POST /api/tasks/suggest`**
- Input: `{ calendar: [events], emails: [emails] }`
- Output: `{ suggestions: [string] }`
- Prompt: ask Claude to extract 3-7 short action items from the calendar and emails. Read CLAUDE.md for role context.

**Ask bar — `POST /api/ask`**
- Input: `{ question: string, calendar: [events], emails: [emails], tasks: [tasks], notion: [pages] }`
- Output: `{ answer: string, sources: [{type, label}] }`
- Prompt: include CLAUDE.md, all the context, and the question. Instruct Claude to cite the source of every factual claim. Refuse anything outside the CLAUDE.md "Out of scope" rule.

## Hard rules

1. **The Anthropic API key NEVER appears in code.** It's always read from `process.env.ANTHROPIC_API_KEY` via dotenv. If you ever see yourself about to write a key inline, stop.

2. **Always use `claude-sonnet-4-6`** as the model identifier. Not Opus (more expensive) and not Haiku (lower quality) unless the task explicitly requires it.

3. **Always set `max_tokens: 1024`** as the default. Higher inflates cost per call.

4. **Every endpoint wraps its logic in try/catch.** On error, return a clean JSON error with HTTP 500 and a user-friendly message. Never leak the actual error message to the dashboard.

5. **Read CLAUDE.md inside every AI call's system prompt.** This is the personalization layer — every reply, brief, suggestion, and answer must reflect the user's profile.

6. **The server never autosends an email.** If a future feature needs to send mail, it can only save drafts. Be paranoid about this.

7. **The server NEVER fetches Gmail or Calendar data directly.** Data display is the frontend-engineer's job (using MCP at generation time). The server only handles interactive AI features.

8. **No `console.log` of full email bodies, full API responses, or anything sensitive.** Short debug logs are fine during build, but never log content.

If the Orchestrator delegates something that isn't server work — HTML, CSS, MCP fetches, visual design — stop and respond:

## Quality bar before handing back

Before you tell the Orchestrator you're done, check:

- [ ] The server starts cleanly with `node server.js` and prints the URL
- [ ] If the API key is missing, the error message is clear and the server exits
- [ ] Every endpoint you added has try/catch
- [ ] Every endpoint that uses Claude reads CLAUDE.md inside the system prompt
- [ ] No secrets are written into code
- [ ] Model identifier is `claude-sonnet-4-6` everywhere
- [ ] No Gmail or Calendar API calls anywhere in the server

If any check fails, fix it before handing back.
